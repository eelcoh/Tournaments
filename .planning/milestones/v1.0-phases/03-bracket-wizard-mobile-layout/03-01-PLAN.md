<objective>
Extend the bracket wizard state and message types to support user-controlled round navigation (stepper taps).

Purpose: The compact 3-step stepper (BRK-01) needs a `viewingRound` field in WizardState so the view can display a completed round the user tapped — separate from `currentActiveRound` which always returns the first incomplete round. Without this state, tapping completed steps in the stepper would have no persistent effect.

Output: Updated `Types.elm` with `viewingRound : Maybe SelectionRound` in `WizardState`, new `JumpToRound SelectionRound` Msg variant, and `update` in `Bracket.elm` handles the new message.
</objective>

**In `src/Form/Bracket.elm`:**

5. The existing `update` pattern-matches on `SelectTeam`, `DeselectTeam`, and `GoNext`. Add a `JumpToRound` handler. When the user jumps to a round, set `viewingRound = Just round` in the wizard state. No bet change occurs — only state updates:
   ```elm
   JumpToRound round ->
       let
           newState =
               { state | bracketState = BracketWizard { wizardState | viewingRound = Just round } }
       in
       ( bet, newState, Cmd.none )
   ```

6. The existing `SelectTeam` and `DeselectTeam` handlers currently set:
   ```elm
   { state | bracketState = BracketWizard { selections = newSelections } }
   ```
   Update these to also carry `viewingRound` forward (preserve it):
   ```elm
   { state | bracketState = BracketWizard { wizardState | selections = newSelections } }
   ```
   This ensures `viewingRound` is not silently dropped when teams are selected/deselected. (Using record update syntax `{ wizardState | selections = newSelections }` naturally preserves all other fields including `viewingRound`.)

7. The `GoNext` handler currently returns `( bet, state, Cmd.none )` unchanged — no update needed there.
