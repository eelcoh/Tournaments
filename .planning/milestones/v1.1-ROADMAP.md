# Milestone v1.1: UX Polish

**Status:** ✅ SHIPPED 2026-03-01
**Phases:** 6–9
**Total Plans:** 7

## Overview

Fixed the group matches scroll wheel stability, added platform-appropriate PWA install prompt banners (iOS and Android), polished the mobile bet form (fixed nav bar, tap feedback, keyboard-primary score input), and eliminated text-selection jank on score keyboard buttons.

## Phases

### Phase 6: Scroll Wheel Stability

**Goal:** The group matches scroll wheel displays reliably — active match anchored, group context always visible, and boundary markers behave predictably
**Depends on:** Nothing (independent)
**Requirements:** SCRW-01, SCRW-02, SCRW-03
**Plans:** 1 plan

Plans:
- [x] 06-01: Rewrite scroll wheel windowing algorithm (7-line window, group label anchoring, END marker, padding lines)

**Details:**
Replaced the old `buildScrollItems` / `viewScrollItem` pair with a strict `buildWindow` function that always produces exactly 7 `WindowLine` items: 3 above + active + 3 below, with `WLPadding` (44px invisible element) filling any short ends. Group label anchoring always replaces `above[0]` with `WLGroupLabel activeGroup` so lines 1–3 stay in the active group's context. `WLEndMarker` is appended after all 48 matches so it can only appear in the below slice (lines 5–7). Eliminates all height jumps and scroll position drift.

### Phase 7: Install Prompt Banners

**Goal:** Players who haven't installed the app are nudged to add it to their home screen via platform-appropriate, dismissable banners that match the terminal aesthetic
**Depends on:** Nothing (independent)
**Requirements:** INST-01, INST-02, INST-03, INST-04
**Plans:** 2 plans

Plans:
- [x] 07-01: JS bridge + Elm ports infrastructure (index.html, Ports.elm, Types.elm, Main.elm)
- [x] 07-02: Banner view rendering in View.elm + human verification

**Details:**
`deferredPrompt` is captured in `<head>` before `main.js` loads and forwarded to Elm via `onBeforeInstallPrompt` port after init. `isIOS` and `isStandalone` are passed as flags (synchronous); `BeforeInstallPrompt` availability arrives via port (asynchronous). `BannerShowingIOS` derived in init from the `isIOS` flag; `BannerShowingAndroid` set when the port fires. A single `inFront` column stacks the install banner above the status bar — avoids z-index issues. Dismiss state persisted via localStorage so banners don't reappear. `BannerHidden` renders `Element.none` — no DOM node, no layout shift.

### Phase 8: Form Mobile Polish

**Goal:** The bet form feels smooth and clear to fill in on a phone — navigation works reliably, incomplete state is obvious, and actions feel acknowledged
**Depends on:** Nothing (independent)
**Requirements:** FORM-01, FORM-02, FORM-03
**Plans:** 2 plans

Plans:
- [x] 08-01: Fixed bottom nav bar (48px, vorige/volgende, per-card incomplete count, ScrollToTop)
- [x] 08-02: Tap feedback

**Details:**
Fixed 48px bottom nav bar with `vorige`/`volgende` greyed (not hidden) at boundaries to avoid layout shifts. Per-card incomplete count shown in the nav bar. `ScrollToTop` fires on every `NavigateTo` as a discard target for `Browser.Dom.setViewport`. Submit button shows "verzenden..." while in-flight. Nav buttons flash white on tap via elm-ui `mouseOver` — maps to CSS `:hover` on mobile, so tap briefly triggers it before navigation. Disabled nav buttons intentionally excluded from mouseOver to keep feedback meaningful.

### Phase 9: Group Match Score Input Improvements

**Goal:** Keyboard-first score input for the 48 group stage matches — flag images always visible in the match header, keyboard as primary entry method with "andere score" link revealing text inputs in-place, and no text selection on keyboard taps
**Depends on:** Phase 8
**Requirements:** (none formally defined — quality improvement)
**Plans:** 2 plans

Plans:
- [x] 09-01: user-select: none on score keyboard buttons (UI/Button/Score.elm)
- [x] 09-02: Keyboard-primary layout + flags in viewInput + andere score overlay (Form/GroupMatches.elm)

**Details:**
`user-select: none` and `-webkit-user-select: none` added via `Html.Attributes.style` to `scoreButton_` leaf function — applied individually to each button cell so mobile doesn't show text-selection highlight on tap. `viewMatchHeader` now always visible (permanent flag row at top of each match view). Default layout shows score keyboard; "andere score" link toggles an overlay showing text inputs with a "terug" link. `ManualInput` boolean in `GroupMatches.State` tracks the toggle. No change to score storage — text inputs still write to `Answer.Score`.

## Milestone Summary

**Key Decisions:**
- Fixed-length windowing: build flat sequence, find cursor index, extract N above + N below, pad with `WLPadding`
- WLPadding renders as 44px invisible element to match match-row height — eliminates layout shift
- Group label anchoring: only replace `above[0]`, never `above[1]`/`above[2]`
- `deferredPrompt` captured pre-main.js to avoid race condition; forwarded via port post-init
- `isIOS`/`isStandalone` as synchronous flags; `BeforeInstallPrompt` as async port message
- Single `inFront` column avoids z-index layering complexity for banner stacking
- `ScrollToTop` as `Task.attempt (always ScrollToTop)` discard pattern
- Greyed (not hidden) disabled nav buttons to avoid layout shifts
- `mouseOver` as tap-flash mechanism — zero state, zero Msg, CSS :hover maps to tap briefly on mobile
- `user-select: none` at leaf (`scoreButton_`) not container for individual button isolation
- `ManualInput` bool in `GroupMatches.State` for score input mode toggle (no separate type needed)

**Issues Resolved:**
- Scroll wheel height jumps at group boundaries → WLPadding fixed height
- Group label disappearing when scrolling past group start → anchoring at line 1
- END marker appearing above active line → restricted to below slice only
- Silent no-op on form submit (no loading feedback) → "verzenden..." state
- Mobile tap shows text selection on score buttons → user-select: none

**Issues Deferred:**
- Live results data integration (LIVE-01, LIVE-02, LIVE-03) → future milestone

**Technical Debt Incurred:**
- Phase 9 requirements not formally documented in REQUIREMENTS.md (added after milestone requirements were finalized)
